# Zero-Trust Network Policies for Stellar Node Isolation

## Overview

The Stellar-K8s operator automatically generates strict `NetworkPolicy` manifests for each managed StellarNode, implementing a zero-trust security model that denies all traffic by default and explicitly allows only required communication paths.

## Architecture

The operator enforces network isolation at **two layers**:

1. **Per-Node Policies** (generated by `src/controller/resources.rs::build_network_policy`)
   - Applied to each individual StellarNode pod
   - Enforces pod-level ingress/egress rules
   - Restricts traffic to only specified ports and peers

2. **Namespace-Level Policies** (deployed via Helm chart `network-isolation.yaml`)
   - Prevents cross-network traffic (e.g., Mainnet ↔ Testnet)
   - Requires namespace labeling: `stellar.org/network=mainnet|testnet`
   - Second layer of defense if pod-level policies are misconfigured

## Per-Node Policy Rules

### Validator Nodes

#### Ingress Rules (Allow)

| Source | Port | Protocol | Purpose |
|--------|------|----------|---------|
| Validators (same network) | 11625 | TCP | SCP consensus protocol (peer-to-peer) |
| Validators (same network) | 11626 | TCP | HTTP admin API (for reconciler health checks) |
| Prometheus namespace | 9090 | TCP | Metrics scraping (if `allow_metrics_scrape: true`) |
| Allowed CIDR blocks | 11625, 11626 | TCP | Optional: Public peers or specific networks |
| Pod selector | 11625, 11626 | TCP | Optional: Specific pod labels (e.g., monitoring sidecars) |

#### Egress Rules (Allow)

| Destination | Port | Protocol | Purpose |
|-------------|------|----------|---------|
| All pods in same stellar.org/network | Any | TCP | Cross-namespace peer communication |
| kube-system/kube-dns | 53 | UDP/TCP | DNS resolution |
| Same namespace pods | Any | TCP | Intra-namespace communication |
| History archive endpoints | 80, 443 | TCP | Ledger archive download (optional) |
| Parsed QUORUM_SET peers | 11625 | TCP | P2P with configured peers (IP-based if available) |

**Default**: All other egress is **DENIED**.

### Horizon & Soroban RPC Nodes

#### Ingress Rules (Allow)

| Source | Port | Protocol | Purpose |
|--------|------|----------|---------|
| Any source (0.0.0.0/0) | 8000 | TCP | Public API endpoint |
| Prometheus namespace | 9090 | TCP | Metrics scraping |

#### Egress Rules (Allow)

| Destination | Port | Protocol | Purpose |
|-------------|------|----------|---------|
| All pods in same stellar.org/network | Any | TCP | Cross-namespace connectivity |
| Stellar Core pods (same namespace, label: `app.kubernetes.io/name=stellar-node`) | 11625, 11626 | TCP | Query ledger state |
| PostgreSQL endpoint | 5432 | TCP | History database (if configured) |
| kube-system/kube-dns | 53 | UDP/TCP | DNS resolution |
| Same namespace pods | Any | TCP | Intra-namespace communication |

**Default**: All other egress is **DENIED**.

## Configuration

### Enable Network Policies per StellarNode

```yaml
apiVersion: stellar.org/v1alpha1
kind: StellarNode
metadata:
  name: testnet-validator
spec:
  networkPolicy:
    enabled: true
    # Optional: restrict ingress to specific namespaces
    allowNamespaces:
      - stellar-testnet
      - stellar-testnet-horizon
    # Optional: restrict ingress to specific pod selector
    allowPodSelector:
      environment: staging
    # Optional: restrict ingress to CIDR blocks (e.g., VPN networks)
    allowCidrs:
      - 10.0.0.0/8
      - 192.168.0.0/16
    # Optional: allow Prometheus metrics scraping
    allowMetricsScrape: true
    metricsNamespace: monitoring
```

### Enable Namespace-Level Isolation

Namespace-level policies prevent cross-network traffic:

```yaml
# values.yaml or helm upgrade
networkIsolation:
  enabled: true
  mainnetNamespaces:
    - stellar-mainnet
    - stellar-mainnet-horizon
  testnetNamespaces:
    - stellar-testnet
    - stellar-testnet-horizon
  allowMonitoringNamespace: true
  monitoringNamespace: monitoring
```

Required: Label your namespaces

```bash
kubectl label namespace stellar-mainnet stellar.org/network=mainnet
kubectl label namespace stellar-testnet stellar.org/network=testnet
```

## Implementation Details

### Default-Deny Behavior

Both ingress and egress policies include their respective policy type in `spec.policyTypes: [Ingress, Egress]`, which activates the Kubernetes default-deny semantics:

- **Ingress**: Any traffic not explicitly allowed by an ingress rule is denied
- **Egress**: Any traffic not explicitly allowed by an egress rule is denied

### Network Isolation Rules

The `build_network_policy` function enforces three mandatory egress rules for all nodes:

1. **Same-Network Rule**: Only allow egress to pods in namespaces with matching `stellar.org/network` label
   - Prevents Testnet pods from connecting to Mainnet resources
   - Supports custom networks via `custom_network_passphrase`

2. **DNS Rule**: Allow egress to `kube-system/kube-dns` on UDP/TCP port 53
   - Required for hostname resolution
   - Explicitly allows DNS for both validators and Horizon nodes

3. **Intra-Namespace Rule**: Allow egress within the same namespace
   - Permits Horizon → Stellar Core communication
   - Permits Soroban RPC → Captive Core communication

### Horizon-to-Core Connectivity

When a Horizon or Soroban RPC node is deployed in the same namespace as a Stellar Core validator, the policy allows communication via port labels:

```
from: Horizon pod
  to: Stellar Core pods (label match: app.kubernetes.io/name=stellar-node)
  ports: [11625, 11626]
```

This ensures:
- Horizon can query the specific Core backend in its namespace
- Cross-node Horizon pods cannot connect to random Core pods
- Network isolation is maintained across namespaces

### Peer Discovery

For validators, the policy extracts peers from the node config:
- **IP-based peers**: Added as `ipBlock` CIDR rules
- **Hostname-based peers**: Allowed via intra-namespace rule (DNS + pod selector)

```
To address DNS-based peer discovery limitations in NetworkPolicy,
the operator converts hostnames to IPs when available and builds
ipBlock rules for explicit peer-to-peer egress.
```

## Testing & Validation

### Verify Policies Are Applied

```bash
# List policies for a node
kubectl get networkpolicy -n stellar-testnet

# Inspect the policy
kubectl describe networkpolicy testnet-validator-netpol -n stellar-testnet

# Export as YAML
kubectl get networkpolicy testnet-validator-netpol -o yaml
```

### Test Connectivity

```bash
# Exec into a pod
kubectl exec -it <pod> -n <ns> -- bash

# Test ingress (from another pod)
nc -zv <service>.<namespace> <port>

# Test egress
curl -I <external-url>
nslookup <hostname>
```

### Common Issues

#### Horizon Cannot Connect to Stellar Core

**Symptom**: Horizon pods get connection refused errors

**Cause**: NetworkPolicy is denying egress to port 11626

**Fix**: Verify Core pods have label `app.kubernetes.io/name=stellar-node`

```bash
kubectl get pods -L app.kubernetes.io/name,stellar.org/instance
```

#### Cross-Network Traffic Blocked

**Symptom**: Testnet pod cannot reach Mainnet pod

**Cause**: Namespace labels or policy namespace mismatch

**Fix**: Verify namespace labels

```bash
kubectl get ns --show-labels | grep stellar
```

#### DNS Not Working

**Symptom**: Pods cannot resolve hostnames

**Cause**: kube-dns egress rule missing or misconfigured

**Fix**: Ensure `allow_metrics_scrape: true` or add explicit DNS allowance

```bash
kubectl exec -it <pod> -- nslookup kubernetes.default
```

## Security Considerations

### Assumptions & Guarantees

1. **Trust Boundary**: Network policies operate at Layer 4 (TCP/UDP)
   - Do not protect against Layer 7 (application) attacks
   - Use mTLS, ServiceMeshes, or API gateways for application-level security

2. **Pod Escape**: Network policies assume the Kubernetes host is secure
   - A compromised pod can still spoof packets if the host is breached
   - Use PodSecurityStandards and Falco runtime detection

3. **Host Path**: Pods on the same host share the network namespace if not isolated
   - Use container runtime security policies (e.g., Falco)

### Monitoring Compliance

Use Prometheus metrics to monitor policy enforcement:

```promql
# Denied packets (if ingress controller logs them)
rate(iptables_dropped_packets_total[5m])

# Pod connectivity health
up{job="stellar-node"} == 1
```
