# Release Process

This document describes how to cut a release of Stellar-K8s, including versioning conventions,
tagging rules, and the automated steps carried out by the CI pipeline.

---

## Versioning

Stellar-K8s follows [Semantic Versioning](https://semver.org/spec/v2.0.0.html):

```
v<MAJOR>.<MINOR>.<PATCH>[-<pre-release>]
```

| Increment | When |
|---|---|
| `PATCH` | Backwards-compatible bug fixes, documentation-only changes, dependency updates |
| `MINOR` | New backwards-compatible features, new CRD fields (optional), new CLI subcommands |
| `MAJOR` | Breaking changes — CRD field removals, removed CLI flags, changed defaults |

Pre-release versions use `-alpha.N`, `-beta.N`, or `-rc.N` suffixes (e.g. `v1.2.0-rc.1`).

---

## Tagging Conventions

- Tags must start with `v` and match the semver pattern exactly: `vX.Y.Z` or `vX.Y.Z-<pre>`.
- The tag version must match the `version` field in `Cargo.toml` exactly (without the `v` prefix).
- Tags are created on `main` only. Never tag a branch or a commit not yet merged.
- Annotated tags are preferred over lightweight tags:
  ```bash
  git tag -a v1.2.3 -m "Release v1.2.3"
  git push origin v1.2.3
  ```

Pushing a `v*.*.*` tag automatically triggers the [release workflow](#automated-pipeline).

---

## Pre-release Checklist

Before creating a tag, verify:

1. **`Cargo.toml` version is updated** — bump `version = "X.Y.Z"` in the workspace root.
2. **`CHANGELOG.md` is updated** — run `make changelog` (uses `git-cliff`) and review the output.
3. **CI is green on `main`** — all jobs in `ci.yml` must pass.
4. **`cargo check` passes** — run `cargo check --workspace` locally.
5. **No uncommitted changes** — run `git status` to confirm a clean tree, or use the CLI:
   ```bash
   stellar-operator health-check
   ```
6. **Version consistency verified** — the CI `version-consistency` job validates this
   automatically, but you can check manually:
   ```bash
   grep '^version = ' Cargo.toml | head -1
   ```

## Automated Pipeline

The `release.yml` workflow runs when a `v*.*.*` tag is pushed. It performs the following steps
in dependency order:

```
validate ──┬── build-artifacts (4 targets: linux-amd64, linux-arm64, darwin-amd64, darwin-arm64)
           ├── container (re-tags sha image from multiarch-build.yml, or builds fresh)
           ├── helm (bumps chart version, packages .tgz)
           └── security (Trivy scan on published image, uploads SARIF)
                └── release (creates GitHub Release, uploads all artifacts + SBOM)
```

### `validate` job

- Extracts the version from the git tag.
- Confirms the version matches a valid semver pattern.
- Confirms `Cargo.toml` version equals the tag version.
- Sets `is_prerelease=true` for tags with a pre-release suffix.

### `build-artifacts` job (matrix)

Builds the `stellar-operator` binary for four targets:

| Target | OS | Artifact name |
|---|---|---|
| `x86_64-unknown-linux-gnu` | ubuntu-latest | `stellar-operator-linux-amd64` |
| `aarch64-unknown-linux-gnu` | ubuntu-latest | `stellar-operator-linux-arm64` |
| `x86_64-apple-darwin` | macos-latest | `stellar-operator-darwin-amd64` |
| `aarch64-apple-darwin` | macos-latest | `stellar-operator-darwin-arm64` |

Each artifact is stripped, tarred (`.tar.gz`), checksummed (`.sha256`), and signed with
[SLSA provenance attestation](https://slsa.dev/) via `actions/attest-build-provenance`.
A CycloneDX SBOM is generated for each binary.

### `container` job

Re-tags the multi-arch image published by `multiarch-build.yml` for the `sha-<commit>` label.
Falls back to a fresh `docker buildx` build if the sha image is unavailable.

Published tags for a full release (non-pre-release):

```
ghcr.io/otowoorg/stellar-k8s:X.Y.Z
ghcr.io/otowoorg/stellar-k8s:X.Y
ghcr.io/otowoorg/stellar-k8s:X
ghcr.io/otowoorg/stellar-k8s:latest
```

For pre-releases (`-alpha`, `-beta`, `-rc`), the `X`, `X.Y`, and `latest` tags are **not** pushed.

### `helm` job

- Updates `version` and `appVersion` in `charts/stellar-operator/Chart.yaml` to the release version.
- Runs `helm lint`.
- Packages the chart as `stellar-operator-X.Y.Z.tgz`.

### `security` job

- Runs [Trivy](https://github.com/aquasecurity/trivy-action) on the published container image.
- Findings are uploaded as SARIF to the repository's Security tab.
- A CRITICAL or HIGH finding does **not** block the release but is visible for review.

### `release` job

- Downloads all artifacts from prior jobs.
- Generates a changelog excerpt with `git-cliff --latest`.
- Creates a GitHub Release with:
  - Binary tarballs and checksums
  - Helm chart tarball
  - Container and binary SBOMs (CycloneDX JSON)
  - A combined `SHA256SUMS` file

## Changelog

The changelog is generated by [git-cliff](https://git-cliff.org/) using `cliff.toml`.
Commit messages must follow [Conventional Commits](https://www.conventionalcommits.org/):

| Prefix | Changelog section |
|---|---|
| `feat:` | Added |
| `fix:` | Fixed |
| `perf:` | Performance |
| `refactor:` | Refactored |
| `doc:` | Documentation |
| `test:` | Testing |
| `chore:` | Miscellaneous |
| `chore(release):` | Skipped (internal) |
| Body contains `security` | Security |

To preview the changelog for the next release:

```bash
make changelog
# or
git cliff --unreleased
```

To update `CHANGELOG.md` with all history:

```bash
git cliff -o CHANGELOG.md
```

## Hotfix Releases

For urgent fixes to an already-released version:

1. Check out the release tag: `git checkout v1.2.3`
2. Create a hotfix branch: `git checkout -b hotfix/v1.2.4`
3. Apply the fix and commit with `fix:` prefix.
4. Update `Cargo.toml` to `1.2.4`.
5. Update `CHANGELOG.md`.
6. Merge to `main` if the fix also applies there.
7. Tag `v1.2.4` on the hotfix commit and push.

---

## Manual Recovery

If the automated release pipeline fails partway through:

- **`build-artifacts` failed for one target**: Re-run the job. Artifacts are uploaded per-target
  and the `release` job re-downloads them all.
- **Container re-tag failed**: Check that the `sha-<commit>` image exists in GHCR. If not,
  the fallback build path will run on re-trigger.
- **Release already partially created**: Delete the GitHub Release draft and the tag, fix the
  issue, re-push the tag.
