# Stale Documentation Detector

> **TL;DR** — When you change a source file that is linked to a doc, the doc
> is flagged as _stale_.  CI will fail.  Update the doc (or suppress the false
> positive), then push.

---

## Overview

Stellar-K8s ships an automated stale-documentation detector (`doc-check`)
that prevents documentation from drifting out-of-sync with the code it
describes.  It works by:

1. Reading `doc-coverage.toml`, which maps every documentation file to the
   source files it covers.
2. Using `git log` to find the most recent commit that touched each source
   file and each doc file.
3. Flagging a doc as **stale** when any of its linked source files was
   committed _after_ the doc's most recent commit.

The tool runs:

| Trigger | Mode | Effect on failure |
|---------|------|-------------------|
| `git push` (pre-commit hook) | warn-only | prints a warning, does **not** block the push |
| Pull-request (GitHub Actions) | scoped to changed files | **blocks merge** |
| Push to `main` (GitHub Actions) | full repo scan | **blocks merge** |
| `make ci-local` | full repo scan | **exits non-zero** |
| `make check-stale-docs` | full repo scan | **exits non-zero** |

## Quick Start

```bash
# Check for stale docs right now
make check-stale-docs

# List all tracked doc → source mappings
make list-doc-coverage

# After updating a doc to make it current, record the new baseline
make update-doc-baseline
```

## Understanding the Output

```
🟡  Stale documentation files:
    STALE  docs/health-checks.md  (src @ a1b2c3d4, doc @ 00112233)
             ↳ src/controller/health.rs

  Summary: 1 stale, 0 missing, 22 ok, 0 ignored
```

| Symbol | Meaning |
|--------|---------|
| ✅ | All tracked docs are up-to-date |
| 🟡 `STALE` | A source file was committed after its linked doc |
| 🔴 `MISSING` | The documentation file does not exist on disk yet |
| (no symbol) | `ignored = true` in `doc-coverage.toml` |

The SHAs printed are the first 8 characters of the Git commit hash.  You can
inspect the full commit with `git show <sha>`.

## Adding a New Mapping

Open `doc-coverage.toml` and add a new `[[entry]]` block:

```toml
[[entry]]
doc = "docs/my-feature.md"
src = [
    "src/my_feature.rs",
    "src/controller/my_feature_helper.rs",
    # Directories expand to all files inside them recursively:
    "examples/my-feature",
]
```

Rules:
- `doc` — relative path to the Markdown file from the repo root.
- `src` — list of file paths or glob patterns (same syntax as `.gitignore`).
- Both paths are resolved relative to the repository root.

Commit both the new code and the new `doc-coverage.toml` entry at the same
time so the baseline stays consistent.

## Handling False Positives

Sometimes a source file changes in a way that does **not** require a doc
update (e.g. a refactor with no user-visible change, a test fix, etc.).

### Option 1 — Update the baseline (preferred)

After reviewing that the doc is genuinely still accurate, record the current
state as the new baseline:

```bash
make update-doc-baseline
```

This writes `.doc-hashes.toml` with the current HEAD SHA for every entry.
Commit the updated baseline file alongside your source change.

### Option 2 — Temporarily silence an entry

Set `ignore = true` on the entry in `doc-coverage.toml` and open a follow-up
issue to update the doc:

```toml
[[entry]]
doc = "docs/complex-feature.md"
src = ["src/complex_feature.rs"]
ignore = true  # TODO: issue #9999 — update this doc
```

Do **not** leave entries permanently silenced; the ignore flag is meant as a
short-term escape hatch only.

## How It Works (Technical Detail)

```
doc-coverage.toml
      │
      ▼
 doc-check binary
      │
      ├─ For each [[entry]]:
      │     ├─ Resolve all src globs to concrete file paths
      │     ├─ git log --follow -1 <src>  →  src_sha
      │     ├─ git log --follow -1 <doc>  →  doc_sha
      │     └─ if src_sha is a descendant of doc_sha → STALE
      │
      ├─ (opt) --changed-files filter: only evaluate files changed in this PR
      │
      └─ Report + exit code
```

### Baseline File (`.doc-hashes.toml`)

The baseline file records the `src` commit SHA at the time each doc was last
marked in-sync.  It is used to detect **regressions** — situations where the
doc was in sync at some earlier point but has since fallen behind again due to
new source commits.

The file is auto-generated by `make update-doc-baseline` and should be
committed to the repository.

## CI Integration

The `stale-docs` workflow (`.github/workflows/stale-docs.yml`) runs
automatically.  It has two modes:

**Pull requests** — only the files that changed in the PR are examined.  This
is fast and avoids noise from pre-existing stale docs unrelated to the PR.

**Push to `main` and manual triggers** — the full repo is scanned.  This acts
as a safety net and surfaces any drift that might have accumulated.

The workflow can be run manually from the GitHub Actions UI with an optional
`warn_only=true` input to get a report without blocking.

## Makefile Targets

| Target | Description |
|--------|-------------|
| `make check-stale-docs` | Run the detector (exits non-zero on any stale/missing doc) |
| `make update-doc-baseline` | Update `.doc-hashes.toml` to current HEAD |
| `make list-doc-coverage` | Print all doc → source mappings and exit |

All three targets are wired into `make ci-local`, so they run as part of the
standard local CI suite.

---

## Pre-commit Hook

The `check-stale-docs` pre-commit hook runs automatically on every commit that
touches a source, doc, or config file.  Locally it uses `--warn-only` mode so
it never blocks your work, but it will print a reminder if a stale doc is
detected.  CI enforces the hard failure.

To run the hook manually across all files:

```bash
pre-commit run check-stale-docs --all-files
```

---

## Adding `doc-check` to a New Workflow

```yaml
- name: Check stale docs
  run: cargo run --locked --bin doc-check -- status
```

To restrict to a specific set of changed files (e.g. in a PR):

```yaml
- name: Check stale docs (scoped)
  run: |
    cargo run --locked --bin doc-check -- \
      --changed-files src/controller/health.rs \
      --changed-files docs/health-checks.md
```

## Troubleshooting

**"Cannot read config file doc-coverage.toml"**
Run `doc-check` from the repository root, or pass `--config path/to/doc-coverage.toml`.

**"All docs are stale after a rebase"**
A rebase rewrites commit SHAs.  Run `make update-doc-baseline` after the
rebase to re-anchor the baseline to the new SHAs.

**"The tool reports a file as stale but I just updated the doc"**
Make sure you committed the doc update.  Uncommitted changes are not visible
to `git log`.

**"I want to check a specific doc only"**
Pass the doc's linked source file to `--changed-files`:

```bash
cargo run --bin doc-check -- --changed-files src/controller/health.rs
```
